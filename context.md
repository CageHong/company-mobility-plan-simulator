## 1. 專案背景

**這是什麼：** 一個互動式網頁模擬器，協助分析公司每年 €100,000 的員工通勤預算。
用來模擬三種補助策略（腳踏車、大眾運輸、兩者並行），即時呈現預算影響、停車位節省與各區員工分布。

**課程背景：** UHasselt 課程 WS4（指導教授：Mohammed Hajouj）的作業，同時作為個人 Portfolio 展示。

## 2. 資料結構

### 來源檔案（放在 `/data/`）

| 檔案名稱 | 原始檔案 | 筆數 | Join 鍵 |
|---------|---------|------|---------|
| `employees.csv` | Employee_data.csv | 1,161 | ID |
| `ttimes.csv` | TTimes.csv | 1,161 | ID |
| `ptcosts.csv` | PTcosts.csv | 1,161 | ID |

### 欄位說明

**employees.csv**
| 欄位 | 型別 | 說明 |
|------|------|------|
| ID | int | 員工編號（1–1161） |
| zone | string | 居住區域（14 個區：ZG、LU、ZH、SZ、AG、NW、UR、OW、SG、GL、BE、SO、BL、SH） |
| PP | 0/1 | 1 = 員工目前有停車位 |

**ttimes.csv**
| 欄位 | 單位 | 說明 |
|------|------|------|
| Ttime_Car (s) | 秒 | 開車通勤時間 |
| Ttime_PT (s) | 秒 | 搭大眾運輸通勤時間 |
| Ttime_bike (s) | 秒 | 騎自行車通勤時間 |
| Distance (m) | 公尺 | 家到公司距離 |

**ptcosts.csv**
| 欄位 | 單位 | 說明 |
|------|------|------|
| Daily ticket Full Price | € | 單日票價 |
| Month pass_Cost | € | 月票價格 |
| Year pass_Price | € | 年票價格（各員工依居住區域不同，€567–€3,860） |

### Master dataset（由 `src/data/loader.js` 產生）
三份 CSV 以 ID 做 inner join，合併成單一物件陣列：
```js
{ id, zone, pp, tCar, tPT, tBike, distance, costDaily, costMonthly, costYearly }
```
程式其他地方只能從這個陣列取資料，不得再接觸原始 CSV。

## 3. 商業邏輯與假設

### 預算結構
| 項目 | 金額 |
|------|------|
| 總預算 | €100,000 / 年 |
| 基準停車費（600 個車位 × €100） | €60,000 |
| 額外可用於新投資的預算 | €40,000 |
| 每個停車位單位成本 | €100 / 年 |

### 腳踏車補助
- **資格條件：** `500m ≤ 距離 ≤ 滑塊設定值`（範圍 500m–5,000m）
- **成本：** €150 / 人 / 年（腳踏車 + 基礎設施：安全停放、充電設備）
- **假設：** 每週上班 5 天 × 46 週 = 230 天基準；混合辦公者依實際到班天數調整

### 大眾運輸補助與混合辦公票種邏輯（v1.1 核心更新）

**個人化臨界點公式：**
```
tipping_point(員工) = 年票價格 / 日票價格
```
每位員工的臨界點不同，因為年票依區域從 €567 到 €3,860 不等。
例如，住 ZG 區的員工臨界點 = €567 / €6.6 ≈ 86 天，但住遠區的員工臨界點更高。

**每年實際通勤天數：**
```
commutingDays = (5 - 每週遠距天數) × 46 週
```

**最優票種自動選擇（每位員工個別計算）：**
```
min(日票 × 通勤天數, 月票 × 12, 年票)
```
若通勤天數低於個人臨界點 → 自動切換為日票，What-if 面板顯示節省金額。

### 停車位回收機制
| 情況 | 車位是否回收 |
|------|------------|
| 完全遠距（5天/週） | ✅ 一律回收 |
| 混合辦公（1–4天/週） | ❌ 保留車位（不回收） |
| 全天到班 + 改騎車/搭PT | ✅ 回收 |
| 全天到班 + 繼續開車 | ❌ 不動 |

**設計決策：** 混合辦公者仍需在到班日使用停車位，故不回收。
停車位共享（hot-desking for parking）未納入模型——這是已知限制，已在報告中說明。

### 運具衝突優先權（兩者皆符合資格時）
預設：成本最低者優先（`cost`）。
使用者可透過 UI 切換為「偏好腳踏車」或「偏好大眾運輸」。

## 4. 架構決策

### 為什麼選靜態網站而非 Streamlit
- 沒有冷啟動延遲（對教授 demo 和 Portfolio 都很重要）
- 視覺設計完全自由，不受 Streamlit 預設框架限制
- GitHub Pages 免費、永久、零維護
- 1,161 筆資料在瀏覽器端運算無壓力

### 為什麼選 Plotly.js 而非 Chart.js
- Hover tooltip 原生支援，config 極少
- 堆疊面積圖 + 警戒線在 Plotly 只需幾行，Chart.js 需要大量手工處理
- 三種圖表（長條、圓餅、堆疊）API 一致

### 為什麼不用 npm / build step
- 推上 GitHub 就上線，零配置
- 教授可直接查看原始碼，不需要 Node 環境
- 所有依賴從 CDN 載入（Plotly.js、PapaParse）

### scenarios.js 的遠距員工篩選邏輯（v1.4 更新）
- v1.0 問題：用陣列 index 截斷 → 不穩定
- v1.1 修正：改用員工 ID 的確定性雜湊（uniform 分布）
- v1.4 更新：改為**距離加權篩選**——將員工依距離由遠到近排序，優先將住最遠的員工標記為完全遠距

**設計理由：** 住得遠的員工通勤成本最高，接受遠距工作的動機最強，比均勻分布更符合現實行為模式。

**與政府政策的關係：** 政府希望公司優先雇用住附近的員工（針對未來招募）。本模型處理的是現有員工的行為調整，兩者不衝突。距離加權反而符合政策精神——讓長途通勤者轉為遠距，短途通勤者繼續到班並成為補助對象。

### 三層架構職責
程式碼在成長過程中保持可維護性：
- **資料層** `src/data/` — 換成 API 只改這裡，其他不動
- **邏輯層** `src/logic/` — 純函數，無 DOM 依賴，可獨立單元測試
- **渲染層** `src/components/` — 每個元件獨立可替換

`app.js` 是唯一知道所有層的檔案。

## 6. 本地執行與部署

### 本地測試
```bash
# 不能直接開 index.html（瀏覽器 CORS 會擋 CSV 載入）
# 必須用本地伺服器：
python -m http.server 8080
# 或
npx live-server .
```
開啟瀏覽器 → `http://localhost:8080`

### 部署到 GitHub Pages
1. 把這個資料夾推到 GitHub repo（例如 `mobility-plan`）
2. 進入 **Settings → Pages**
3. Source 選 `Deploy from branch` → `main` → `/ (root)` → Save
4. 約 60 秒後網址出現：`https://<你的帳號>.github.io/mobility-plan`

每次 `git push` 到 `main` 都會自動重新部署，無需任何 CI。

## 7. 已知限制（可寫入報告）

1. **停車位共享未建模：** 混合辦公情境下，A 週一三五到班、B 週二四到班，理論上可共用一個車位。目前模型假設混合辦公者仍佔用獨立停車位，因此低估了停車節省效益。

2. **遠距工作依距離加權，非職位加權：** v1.4 起改為優先將住最遠的員工標記為遠距，比均勻分布更符合現實。但現實中遠距比例也會依職位類型而異（例如行政人員比工廠作業員更容易遠距），這部分因資料中無職位欄位而無法建模。

3. **共乘為估算值：** 共乘人數用百分比滑塊估算，假設每台車平均 2 人，無法追蹤具體配對。

4. **腳踏車補助不考慮到班頻率：** 混合辦公者每週只來 2 天，是否仍值得補助 €150 的腳踏車設備，模型未做差異化處理（可作為未來優化方向）。

5. **靜態模型，不模擬長期行為反應：** 本模型分析現有 1,161 名員工的通勤行為，不考慮員工對政策的長期反應（例如因遠距政策成熟而搬到更遠的地方）。實際執行時建議定期重新調查員工通勤資料，更新模型基準。
